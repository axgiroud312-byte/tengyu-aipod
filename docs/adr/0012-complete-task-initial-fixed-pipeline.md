# ADR-0012 — 完整任务最初版采用内置固定流程

## Status

Accepted, amended by ADR-0013

> 说明：本文记录“完整任务先采用内置固定顺序”的初始决策。后续步骤开关、检测覆盖、macOS 边界和结果区体验以 ADR-0013 为准；本文已按当前口径同步，避免旧表述误导实现。

## Context

用户需要把“采集 / 生图 / 抠图 / 侵权检测 / PS 套版 / 标题生成”按顺序跑起来，
减少跨模块手动切换和重复选目录。

已有模块都具备独立 runner：

- 生图模块负责文生图、图生图、提取和抠图。
- 侵权检测模块负责批量检测和风险分类。
- PS 套版模块负责模板批次输出。
- 标题模块负责读取模板批次并写 `标题.xlsx`。

但通用编排引擎还没有完成。直接在 v1 做自由流程编辑器，会同时引入模板 DSL、
资源队列、暂停恢复、失败策略、断点续跑和上架串接，风险过大。

## Decision

v1 先做 **完整任务最初版**，采用内置固定流程。完整任务不是自由流程编辑器，但任务起点可以选择生成新印花或使用已有印花：

```text
任务起点
  ├─ collection：从采集目录读取原图并提取（Grsai / 晨羽）
  ├─ txt2img：Grsai，先由阿里云百炼写提示词
  ├─ img2img：Grsai
  └─ existing_prints：已有印花来源，可从抠图 / 侵权检测 / PS 套版开始
  ↓
可选抠图
  ↓
可选侵权检测（默认读取检测模块已保存配置，也可为本次运行覆盖模型和 Skill）
  ↓
可选准备等待套版印花（启用 PS 时按印花货号命名）
  ↓
可选 PS 套版
  ↓
可选标题生成（依赖 PS 套版）
```

首版边界：

- 不包含上架；上架仍由上架模块读取 `04-上架工作区` 后单独执行。
- 不做自由流程编辑器。
- 不做暂停 / 恢复 / 断点续跑。
- 只支持取消，且取消是尽力取消。
- 局部印花默认抠图，满印默认不抠图。
- `existing_prints` 只允许选择 `02-印花工作区` 下的具体印花图片文件夹，不允许直接选择 `02-印花工作区` 根目录或 `等待套版` 目录。
- `existing_prints` 用 `startStep: 'matting' | 'detection' | 'photoshop'` 表示从哪里开始，默认从 `photoshop` 开始并默认开启 PS 套版和标题生成；起点之前的步骤锁定为跳过，后续步骤继续按显式开关执行。
- 后续步骤始终使用上一阶段的最新产物：抠图后用抠图结果，检测后用检测通过结果，直接 PS 时用已有印花来源图片。
- 采集+提取可按能力选择 Grsai 或 ComfyUI 路径；完整任务里的 ComfyUI 提取 / 文生图 / 图生图 / 抠图步骤都独立选择运行云机，设置页默认云机只是默认候选。同一完整任务内多个 ComfyUI 步骤选择同一运行云机时，按固定步骤顺序使用。
- 文生图统一先用阿里云百炼生成提示词；随后可选 Grsai 文生图或 ComfyUI 文生图工作流，不保留“自己写提示词”模式。
- 图生图支持 Grsai 和 ComfyUI 两条路径：Grsai 路径沿用参考图、参考方式和“生图时带参考图”开关；ComfyUI 路径选择图片文件夹、工作流、运行云机和每张生成数量。
- 侵权检测可选；`block` 拦截，`pass` / `review` 放行。
- 完整任务页不嵌入完整检测设置面板；默认读取独立侵权检测模块已保存配置，并允许为本次运行覆盖模型、Skill、通过要求和压缩开关。
- 只有启用 PS 套版时，完整任务才必须填写 **印花货号**。进入 PS 前，客户端把最终可套版印花复制到
  `02-印花工作区/等待套版/{runId}/`，命名为 `{清洗后的印花货号}{清洗后的分隔符}{四位序号}.{ext}`，
  序号从 `0001` 开始，仅在本次完整任务内递增。
- PS 套版使用等待套版印花的文件名作为分组 ID，所以输出货号文件夹与等待套版印花文件名同名；
  已有印花来源的原始文件名不要求等于最终货号名，也不会被改名或移动；
  默认输出根目录是 `04-上架工作区`。
- macOS 只在启用 PS 套版时禁止启动；关闭 PS 后可以运行前置步骤。

实现上复用已有模块 runner，不模拟 UI 点击，不让服务端参与图片、LLM 或 API Key。
完整任务页采用“上方配置、下方结果”的工作台布局，结果区展示来源图、参考图、印花产物、侵权检测结果、运行摘要和完整任务运行期日志。

运行记录使用独立 SQLite 表：

- `pipeline_runs`
- `pipeline_steps`

这些表是固定完整任务首版的运行记录，不代表 v1.5 通用任务模型已经落地。

## Consequences

### Positive

- 快速满足最常用链路，不需要等通用编排引擎。
- 复用现有模块能力，避免复制业务逻辑。
- 上架仍保持独立模块边界，不影响 ADR-0004 的上架四层结构。
- 后续可以从真实使用反馈决定 v1.5 通用编排模板，而不是提前设计过度。

### Negative

- 首版不能自由增删步骤。
- 首版仍不是自由编排器；后续步骤只能按内置顺序开关，不能任意改顺序。
- 首版失败策略固定，不能配置 `skip` / `pause`。
- 首版运行记录和未来通用任务模型存在一次迁移/兼容成本。

## Follow-ups

- v1.5 再设计通用编排引擎：多模板、资源队列、暂停恢复、失败策略、上架串接。
- 如果用户强烈需要上架串接，先评估上架 profile 锁、店铺选择和失败恢复，再纳入通用编排。
