# ADR-0005 — 货号与任务两层模型；货号 = 一个上架 listing

**状态**：已采纳
**日期**：2026-05-23

## 背景

腾域全流程涉及很多"工作单元"概念：
- 一次采集
- 一次生图
- 一次套版
- 一次上架

用户文档里随口说"一张图就是一个任务 id 就是货号"——但实际场景比这复杂得多：

- 一个商品页有 8 张采集图
- 从中提取 2 个印花
- 一个印花套 3 个模板 = 3 个上架 listing
- 用户可能对同一货号重新跑套版（试不同效果）

需要一个清晰的概念模型表达这种复杂性。

## 决策

### 1. 货号粒度

**货号 = 一个最终上架的 listing**（方案 R）。

| 场景 | 货号数 |
|---|---|
| 8 张采集图 | 0（采集图不是货号）|
| 1 个商品页图池分组 | 0（商品页分组不是货号）|
| 2 个提取的印花 | 0（印花不是货号）|
| 1 个印花套 3 个模板 → 3 条 listing | 3 |

### 2. 模板批次作为外层组织

```
04-上架工作区/
├─ {模板1名}/                   ← 一个模板 = 一个"上架批次"
│   ├─ {印花A}/                 ← 货号文件夹（批次内唯一）
│   ├─ {印花B}/
│   └─ 标题.xlsx                ← 该批次的标题表（兼容旧 titles.xlsx）
├─ {模板2名}/
│   ├─ {印花A}/                 ← 同印花在另一模板下，是不同的 listing
│   └─ 标题.xlsx
```

- **货号在批次内唯一**（用印花名）
- **同印花 × N 模板 = N 个 listing**，分散在 N 个批次目录里
- **上架程序的扫描单位是模板批次目录**

### 3. 两层任务模型

**货号是业务对象**，**任务是工作执行单元**，两者分离：

- 同一货号可以有多个历史任务（重做、补做）
- 同一货号同时刻最多一个"进行中"任务
- 一个任务始终绑定一个货号

**任务分两种类型**：
- **轻量任务 (lightweight)**：单模块独立运行，1 个 step，可能没 sku_code
- **完整任务 (full)**：跨模块多 step。v1 先由内置固定完整任务服务驱动；
  v1.5 再升级为通用编排引擎。sku_code 在套版产生货号后才稳定绑定。

### 4. 货号格式

- 允许：英文字母 + 数字 + `-` + `_`
- 禁用：Windows 非法字符、空格、中文、保留名
- 长度 1-60
- 大小写比较不敏感

## 替代方案对比

| 方案 P：8 个货号（每采集图 1 个） | 简单但冗余 |
| 方案 Q：1 个货号（一个商品页 = 一个货号）| "1 货号 → 3 listing" 难表达 |
| **方案 R：3 个货号（每 listing 1 个，采纳）** | 清晰对应上架 |
| 方案 S：3 层 ID（采集组 + 印花 + 货号）| 概念太多 |

## 实施细节

### 货号生成时机

| 时机 | 生成方式 |
|---|---|
| 套版模块（默认）| 输入印花文件名（清洗后）|
| 套版前未填货号（自动模式）| `{平台}-{日期}-{序号}` |
| 模块独立测试 | `TEST-{时间戳}` |
| 本地批量导入印花 | `{用户前缀}-{序号}` |

### 重做策略

- 同货号重做时**不复制货号**，开新任务（旧任务标记 `'failed'` 或 `'cancelled'`）
- 04-上架工作区/{batch}/{sku}/ 下旧成品图**自动备份**到 `.workbench/archive/{sku}-v1/`
- 用户想保留两个版本对比 → 手动改名为 `SKU001-A` / `SKU001-B`（不自动）

### 改名规则

- 任务进行中：禁止改货号
- 任务完成/已归档：允许改
- 改名联动：文件夹改名 + 数据库 `sku_code` 字段更新 + 标题 xlsx 行键更新

## 影响

### 正面

- 货号粒度与上架平台 1:1 对应，扫描和上架流程清晰
- 任务有完整历史（重做/补做不会丢失数据）
- 模板批次目录天然作为"上架批次"，便于分批上架

### 负面

- 采集阶段没有货号（采集图不是货号）—— 用户对此有心智冲突
- 图池里的“商品页文件夹”只是采集分组，不是最终上架货号
- 用户可能希望"看一个印花在所有模板下的所有结果"——需要数据库视图（通过印花 ID 查询）

## 缓解

- 文档和首次启动引导**清晰说明**"货号 = 上架 listing"
- 客户端 UI 提供"按印花查所有衍生货号"的查询入口（数据库视图）

## 替代决策的触发条件

如果用户反馈"货号和模板批次的双层目录太复杂"，可以考虑：
- 把模板批次"扁平化"（每个货号自带模板后缀，如 `SKU001-template1`）
- 但这会丢失"按批次上架"的便利

这个权衡留 v1.5 看用户反馈再调。
