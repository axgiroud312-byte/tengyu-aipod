# ADR-0015 — 完整任务改为流式流水线

**状态**：已采纳
**日期**：2026-06-27
**决策者**：腾域 aipod 项目

## 背景

ADR-0012 / ADR-0013 把完整任务定为**固定顺序的批处理**：每一步都 `await` 整批完成才进下一步——
生成整批做完 → 抠图整批 → 检测整批 → 套版整批 → 标题整批（`pipeline-service.ts` 的 `runPipeline`）。

真实使用里用户要的是"不停"：一张印花一生成出来，就顺着 抠图 → 检测 → 套版 → 标题 往下溜，
生图那边继续出后面的图。批处理让机器在"等整批齐"的间隙空转，与"一直运行起来"的诉求冲突。

ADR-0012 同时规定 v1 **不做通用编排引擎**（暂停恢复、失败策略、资源队列留 v1.5）。
本决策不引入通用编排引擎，只把"同一批内逐项流动"提进 v1；引擎级的暂停恢复仍留 v1.5。固定完整任务允许用户手动续跑 `interrupted` / `failed` run。

## 候选方案

| 方案 | 优势 | 劣势 |
|---|---|---|
| **A. 流式流水线（逐项流动）** | 机器不停、整体更快、贴合"一直跑"诉求 | 把 v1.5 一部分提前；状态比批处理复杂 |
| B. 保持批处理 | 状态最简单 | 与"不停"诉求冲突，机器空转 |
| C. 通用编排引擎 | 最灵活、支持任意流程 | v1 风险过大（ADR-0012 已否决） |

## 决策

**采纳 A：完整任务改为流式流水线。** 取代 ADR-0012 / ADR-0013 中关于"固定批处理执行模型"的表述；
ADR-0012 / 0013 的其余内容（来源四类、显式开关、任务起点/等待套版命名、UV 工作台布局、不包含上架）仍然有效。

流式流水线的执行契约：

- **流的单位**：套版前是**印花**，套版后**扇出为货号**（印花数 × 模板数）。每张印花一到套版就立刻套所有模板，不等别的印花凑齐。
- **生图不停**：生图整批仍跑完，但函数新增 `onImageComplete` 回调，每张印花算完立即把路径、印花 ID、artifact 血缘吐给流水线，流水线立刻送进下一步。Grsai 与 ComfyUI 两条路都加。
- **云机交错**：废掉 ADR-0012 "同一完整任务内多个 ComfyUI 步骤选同一台云机时按步骤顺序串行"的规矩。同台云机可交错往其队列提交、由 ComfyUI 内部串行；想更快就开两台云机分别负责生图和抠图。
- **检测分流**：被拦（`block`）的印花踢出流水线、归入"未通过"，不堵后面；检测既是处理也是过滤，逐张来。检测全被拦、零货号产出算**正常完成**，仅给警告。
- **失败隔离**：单张印花或单货号在某步出错只标失败并踢出、记日志，不影响其他印花和整条 run。任一阶段整体无法启动或无法安全继续（例如配置或鉴权错误、云机未就绪、Photoshop 未运行、关键业务文件无法持久化）时，整条 run 标记为 `failed`；用户软停标记为 `cancelled`。其余都是"部分失败但 run 正常完成"。
- **序号先到先得**：等待套版印花的四位序号按**到达套版的先后顺序**发放（从 `0001` 起、本次 run 内递增），不按生图提交顺序预分配。
- **标题逐货号**：每个货号一套版完成就立刻生成标题、累加写进该模板批次 `标题.xlsx`；xlsx 被 Excel 占用时先把结果暂存、待可写时补写，不阻塞流水线、不报错。
- **取消软停**：不再接新活（生图停、未开始的印花/货号不再往下放），已在跑的单张让它跑完，不强制中断外部 API / Photoshop；已完成的结果保留。
- **不自动续跑**：软件中途关闭后不自动续跑——该 run 标"已中断"，已生成的印花、套好的版、写好的标题保留在硬盘和数据库。用户可手动续跑 `interrupted` / `failed` 的固定完整任务，或从"已有印花来源"起新 run；通用编排引擎级断点续跑仍留 v1.5。
- **结果区**：按阶段分栏竖排、各栏同时流动，复用现有 `RESULT_SECTION_ORDER` 机制；印花产物区沿用"完成一张显示一张"。

## 影响

### 正面

- 机器不停、整体吞吐更高，贴合用户"一直运行起来"的核心诉求。
- 失败隔离让单张出错不再拖垮整批，体验更稳。
- 仍不引入通用编排引擎，v1 边界（不自由编排、不含上架）不变。

### 负面

- 把 v1.5 的一部分（逐项流动、单项失败隔离）提前到 v1，`pipeline-service` 的 `runPipeline` 要从"阶段串行 await"重写为"生产者-消费者流"。
- 生图函数签名要加回调参数，`generation-service` 所有 run 函数及调用方都要改。
- `pipeline_steps` 记录要从"整批边界更新"改为"逐项进度"，运行记录表结构需扩展。
- 同台云机交错排队可能在用户机器偏弱时增加显存压力；缓解方式是引导开两台云机。

## 与 ADR-0012 / ADR-0013 的关系

本 ADR 取代两份 ADR 中**执行模型**相关表述：批处理阶段串行、同机按步骤顺序串行、"生图批处理主要在 step 边界停止"的取消语义、整批失败语义。
两份 ADR 的**配置与边界**内容（来源四类、显式开关、`startStep`、等待套版命名、UV 工作台、不包含上架、macOS 限制）继续有效。

## Follow-ups

- 实现：重写 `pipeline-service.ts` 的 `runPipeline` 为流式；`generation-service` 加 `onImageComplete`；`pipeline_steps` 扩展逐项进度。
- v1.5 通用编排引擎仍负责：多模板、资源队列、暂停恢复、通用断点续跑、上架串接。
