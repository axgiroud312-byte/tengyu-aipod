## 1. 产品定位

**腾域 aipod** 是面向**中国跨境电商运营**的桌面工作台，把"采集产品图 → 生成印花 → 侵权检测 → PS 套版 → 标题生成 → 店小秘批量上架"全流程串成一站式工具。

**目标用户**：跨境电商运营（POD 印花品类为主）。月流水 1-50 万人民币的中小卖家。

**核心痛点**：
- 全流程分散在多个工具里，跨工具迁移文件耗时
- 提示词、检测规则、上架字段全靠人工配置，不能复用
- 没有持续更新的"内置专家经验"

**解决方案**：把全流程集成在一个桌面客户端，配合云端持续派发提示词（Skill）、ComfyUI 工作流、上架机器人选择器更新，让用户始终用上"最新版业务经验"。

## 2. 用户故事

### Story A：从 0 到上架（全链路）

> 跨境运营**老王**在 Temu 上看到一款热销 T 恤。他用腾域**采集**了商品页的 8 张产品图；切到**生图模块**选 ComfyUI 提取工作流，从其中 2 张图提取出 2 个印花；进**侵权检测**，2 个都判定低风险通过；选 3 个不同的 PSD 模板（白T 正面 / 黑T 背面 / 卫衣正面）做**套版**，得到 6 个货号文件夹；**标题模块**给 6 个货号生成英文标题；最后**上架**到他的 2 个 Temu 店铺，几分钟搞定 12 条 listing。

### Story B：手里有印花直接套版（半链路）

> 设计师**小李**自己画了 20 张印花。她跳过采集和生图，把印花直接拖入腾域的**04-待套版印花**目录，**套版模块**选 5 个模板做笛卡尔积输出（100 个货号），跳过侵权检测，**标题模块**生成西班牙语标题（卖 Mercado Libre），**上架**完事。

### Story C：纯生图试验（不上架）

> 个人卖家**小张**还在摸索风格。他用**生图模块的付费 Grsai** 反复试不同提示词风格的圣诞印花，跑出 50 张候选，存在 02-生图 下。这周不打算上架，攒一批等下周决定再走后续流程。

### Story D：用户管理（你视角）

> 你（管理员）在浏览器打开 admin 后台：
> 1. 给新客户**小赵**手动建一个激活码（365 天，1 设备）
> 2. 给老客户**王总**追加 30 天（他续费了）
> 3. 批量生成 100 个匿名试用码发到推广群
> 4. 上线新版本 v1.4.0，编辑 changelog，发布
> 5. 修改"标题生成 - Temu - 英文"的 skill 提示词，所有客户端 30 分钟内自动拉到新版本
> 6. 监控错误日志：发现"店小秘 SELECTOR_NOT_FOUND"上报集中爆发，推一次上架机器人选择器更新

│       │            │            │            │              │

└─────────────────────────────────────────────────────────────┘
       │                  │                      │
       │ 直连             │ 直连                │ 联网
       ▼                  ▼                      ▼
┌──────────────┐  ┌──────────────┐  ┌──────────────────────┐
│ 比特浏览器   │  │ 晨羽智云     │  │ 腾域云服务器          │

**核心原则**：
1. 业务计算和用户数据**全在本机**
2. 云端**只派发配置**，不代理任何用户数据
3. 模块按"业务能力"组织（而非按 provider）
4. 跨模块共享的能力（比特浏览器、CDP、临时文件管理）放 adapter 层

## 6. 货号与任务模型

### 货号粒度

**货号 = 一个上架 listing**（方案 R）。

| 输入 | 货号数 |
|---|---|
| 8 张采集图 | 0（采集图不是货号）|
| 2 个提取的印花 | 0（印花不是货号）|
| 印花 A 套白T 正面 = 1 个 listing | 1 |
| 印花 A 套黑T 背面 = 1 个 listing | 1 |
| 印花 B 套白T 正面 + 黑T 背面 = 2 个 listing | 2 |
| **合计** | **3** |

**模板批次** = `05-货号成品/{模板名}/`，每个模板自成一个上架批次目录，对应一个 titles.xlsx。

### 货号格式

- 允许字符：英文字母（大小写）、数字、`-`、`_`
- 禁用：Windows 非法字符、空格、中文、保留名（CON/PRN/AUX/...）
- 长度 1-60
- 大小写比较不敏感（避免文件夹冲突）

### 货号生成策略

| 场景 | 货号生成 | 例 |
|---|---|---|
| 套版模块（默认）| 输入印花文件名（清洗后）| `pri_001` → 货号 `pri_001` |
| 套版前未填货号（自动模式）| `{平台}-{日期}-{序号}` | `TEMU-20260523-0001` |
| 模块独立测试 | `TEST-{时间戳}` | `TEST-20260523-153045` |
| 本地批量导入印花 | `{用户前缀}-{序号}` | 用户填 `BATCH1-` → `BATCH1-0001` |

### 任务两层

- **完整任务**：跨多模块，由编排引擎驱动（v1.5）
- **轻量任务**：单模块独立运行，每次执行也建任务（v1）

任务状态：`未开始` → `进行中` → `完成` / `失败` / `已弃 (重做时旧任务标记)`
同一货号同一时刻**最多一个"进行中"任务**。

## 7. 模块详细需求

详细 spec 见 `spec/` 目录，下面只列每个模块的 v1 核心需求。

### 7.1 采集模块

**v1 内置 7 个平台**：Temu / Ozon / Shein / TikTok / Shopee / 1688 / Mercado Libre

**核心模式**：
- **图池扫描**：用户在比特浏览器打开搜索页/列表页/商品详情页，腾域扫描当前页图片索引，用户勾选后下载。
- **商品详情主图采集**：商品详情页只采集左侧主图/轮播图，按商品页文件夹展示和保存。
- **点击采集**：用户点击图片保存原图到 `01-采集/{货号}/`（货号在该模式下可选）。
- **滚动采集**：用户滚动列表页，按规则自动保存图片到散图池。
- **运行期日志**：采集页提供命令行式日志弹窗，实时显示扫描、下载、点击采集和会话诊断。

**约束**：
- 同时刻 workbench 内最多 1 个采集会话
- 一个会话只绑定一个**浏览器 profile**（与上架互斥）
- 不绕过登录/验证码/风控（用户自己在比特浏览器登录）
- 图池下载先在扫描阶段标记“散图/商品页”归属，保存和预览都按该归属展示。

详见 [spec/02-collection.md](./spec/02-collection.md)。

### 7.2 生图模块（统一模块按能力分 4 Tab）

**4 个能力 Tab + 2 个 Provider**：

| 能力 | 付费（grsai）| ComfyUI 免费（comfyui-chenyu）|
|---|---|---|
| 文生图 | ✅ | ❌（comfyui 不做文生图）|
| 图生图 | ✅（5 种模式：纯文字/参考构图/参考风格/构图+风格/自己写）| ✅（多个工作流可选）|
| 提取 | ✅（图生图 + 提取 skill）| ✅（提取工作流）|
| 抠图 | ❌（不内置）| ✅ ComfyUI 抠图工作流 / "黑白图 + 转遮罩混合"路径 |

**用户视角**：选 Tab（能力）→ 选实现方式（付费/comfyui）→ 配置 → 跑。

**关键设计**：
- **付费模式下"图生图"5 种生成方式**：纯文字 / 带参考图（参考构图）/ 带参考图（参考风格）/ 带参考图（构图+风格）/ 自己写提示词
- **Skill 只管输入变量**，输出格式由 skill 自约束（JSON / 分隔符），客户端通用解析器兜底
- **抠图无纯付费路径**（付费模型抠不好），只有 comfyui 工作流或混合路径

**晨羽智云集成**：
- 用户填 API Key（OS keychain）
- 客户端管理实例：创建/启动/查状态/连接 ComfyUI 8188 端口
- 默认设定时关机（晨羽侧定时关机 API），软件崩了不漏关
- 客户端调 ComfyUI 原生 HTTP API（不走晨羽 workflow/run 接口）
- 工作流包从腾域云服务器派发

**Grsai 集成**：
- 用户填 API Key
- 直接调 `/v1/api/generate`（统一接口 + model 字段切换）
- 支持 11 个模型（nano-banana 全系列 + gpt-image-2）
- 支持 base64 参考图、宽高比、分辨率
- 三种 replyType：json（同步）/ stream（流式）/ async（异步轮询）
- 国内 / 全球节点自动选择

详见 [spec/03-generation.md](./spec/03-generation.md)。

### 7.3 侵权检测

**Provider**：阿里云百炼

**模型选择**（用户可选）：
- qwen3-vl-flash（默认，便宜：输入 ¥0.15/百万Token）
- qwen3-vl-plus（平衡）
- qwen-vl-max（旗舰）

**风险三档**：
- 0-39 → `03-检测/pass/`
- 40-69 → `03-检测/review/`
- 70-100 → `03-检测/block/`
- **阈值用户可调**

**输入预处理管线**：
- 透明底加白（强制）
- 压缩（可选，默认 1024px JPG q=85）
- 用 `sharp` 库，并发受配置档限制
- Worker Thread 跑不卡 UI
- 临时文件由 TempFileManager 管理

**Skill 输入变量**：模型、是否输出依据、关注重点（多选）

**下游**：用户可一键流转 pass/ 的图到 `04-待套版印花/`

详见 [spec/04-detection.md](./spec/04-detection.md)。

### 7.4 PS 套版（最复杂，Windows only）

**多模板批次输出**：用户选印花文件夹 + 选 N 个模板 → 输出到 `05-货号成品/{模板1}/`、`05-货号成品/{模板2}/`…

**PSD 模板扫描**：按 hash 缓存，扫描结果包括智能对象/参考线/裁切区/嵌套关系。

**JSX 实现两条路径**：
- **路径 A（v1）**：直接 `placedLayerReplaceContents`，简单可靠（xKeNcHii 极简模式）
- **路径 B（v1.5）**：进入 SO 编辑 + flattenImage + Plc，处理嵌套/共享 SO（joonaspaakko 复杂模式）

**v1 检测嵌套 SO 时**：UI 提示"v1 不支持，效果可能不理想"。

**UI 输入**：
- 印花文件夹路径
- 模板（多选）
- 替换范围（自动/顶层/全部）
- 适配方式（fill/fit/center）
- 导出格式（JPG/PNG）
- 裁切模式（none/auto/guides）
- 跳过已完成、失败重试

**进度面板**：总分组、完成、失败、跳过、已验证输出。

详见 [spec/05-photoshop.md](./spec/05-photoshop.md)。

### 7.5 标题生成

**Provider**：阿里云百炼

**用户在 UI 填**：
- 货号批次路径（`05-货号成品/{模板批次}/`）
- 平台（云端拉支持列表，含 Temu/Shopee/Amazon/...）
- 语言（云端拉支持列表）
- 模型（用户可选 qwen3-vl-flash / plus / max）
- 取第几张图（默认 1）
- 标题额外要求（textarea）
- 已有标题策略（跳过 / 重新生成）

**输出**：
- 在批次目录写入 `titles.xlsx`（货号 | 标题，两列）
- 失败的货号留空 + UI 列出失败清单

详见 [spec/06-title.md](./spec/06-title.md)。

### 7.6 上架

**Provider**：店小秘（通过 Playwright + 比特浏览器 CDP）

**v1 支持的平台**：
- Temu PopTemu ✅
- Shein ✅
- 其他 4 个平台（TikTok / Temu Full / Ozon / Mercado）→ v1.5

**核心策略**：
- **直接 Port** `一键pod/上架程序` 的**框架代码**（runner.ts 编排、CDP 连接、状态机、错误码）
- **重写**店小秘 12 阶段 DOM 操作代码（按 listing-automation-builder SKILL 规范的四层结构：selectors / page-parser / action-executor / workflow）
- **每个平台独立目录**，按 SKILL 严格分层
- **关键资产**：用户的店小秘"草稿模板"预存价格/类目/规格/运费，腾域只覆写标题/SKU/图片

**UI 输入**：
- 货号批次目录
- 平台
- 比特浏览器 profile（多选，工作区）
- 草稿模板 ID（用户手填）
- SKU 编码策略（自动生成 / xlsx 列）
- 提交方式（保存草稿 / 直接发布）
- 高级：每店铺并发 / 失败重试 / 连续失败暂停阈值
- ✅ 启用断点续传

**断点续传**：数据库记录每个 `(批次, 货号, 平台, profile)` 的 listing_status，重启自动跳过 success。

**与采集互斥**：同一 profile 不能同时被采集和上架占用，由 BrowserProfileLock 协调。

详见 [spec/07-listing.md](./spec/07-listing.md)。

## 8. 编排（v1.5）

v1 不做编排引擎，每个模块独立使用产生轻量任务。v1.5 引入：

**6 个内置流程模板**（按 Q6 修订）：
1. 完整链路：采集 → 提取 → (图生图) → 抠图 → 检测 → 套版 → 标题 → 上架
2. 从印花开始：本地导入 → 检测 → 套版 → 标题 → 上架
3. 只生图：采集 → 提取 → 检测 → 停在 04
4. 只上架：05 已有成品 → 标题 → 上架
5. 只套版：选印花 → 套版 → 停在 05
6. 套版加上架：选印花 → 套版 → 标题 → 上架（跳过检测）

**两种执行模式**：自动连跑 / 逐步确认

**失败传播默认策略**：任一步失败 → 停止整个任务（用户可改）

详见 [spec/01-orchestration.md](./spec/01-orchestration.md)。

## 9. 服务器端

**技术栈**：Next.js + Postgres + Prisma + shadcn/ui admin

**职责严格边界**：
- ✅ 激活码管理、客户管理

## 10. 安全模型

**总原则**：完美安全不存在，但腾域有天然优势：
- 核心资产（Skill / 工作流 / selectors）在服务端
- 业务依赖持续更新（破解版迅速过时）
- 目标用户是商人，盗版动力弱
- 跨境圈小众，破解版传播链有限

**v1 必做的 7 项基础防御**：
1. HTTPS + JWT 签名（防抓包改包）
2. 设备指纹哈希（防多机滥用）
3. 时钟回拨检测（防离线绕过期）
4. 强制 7 天联网周期
5. 关键操作（生图、上架）必须实时联网验证
6. JS 代码混淆（JavaScript Obfuscator）
7. 关键逻辑在服务端（激活验证、剩余天数计算）

**v1.5+ 增强**：bytenode 编译、多点交叉验证、服务端风控、代码完整性校验。

## 13. 错误、日志、遥测

- **日志**：`.workbench/logs/`，结构化 JSON，30 天自动清理
- **采集运行期日志**：采集页弹窗内存保留最近 `1000` 条，不落盘；用于当场排查扫描/下载进度。
- **崩溃日志**：单独目录，下次启动询问是否上报
- **错误上报**：默认开启 + 可关，只上报错误类型/堆栈/模块/版本（不上报用户数据/图片/Skill 内容）
- **服务端聚合**：admin 后台看错误分布，发现集中爆发推热修复

## 14. 并发与资源管理

**全局并发上限**：默认 3，范围 1-10，用户可调

**资源互斥**：
- 比特浏览器 profile：BrowserProfileLock（采集 vs 上架）
- PS 实例：同时刻 1 个套版任务
- 晨羽实例：单实例多任务 OK（ComfyUI 内部队列）

**任务队列**：主进程内 priority queue，优先级 = 用户手动 > 编排引擎自动 > 重试

**暂停恢复**：按模块粒度
- 采集：会话级 / 图池级（图池扫描结果在前端内存中，重启后重新扫描）
- 生图：单图级
- 检测：单图级
- 套版：任务组级
- 标题：单货号级
- 上架：单 listing 级
