# ADR-0010 — 采集图池先分类；运行期日志先内存化

**状态**：已采纳
**日期**：2026-05-29

## 背景

采集模块需要支持三种真实使用方式：

- 搜索页/列表页一键扫描商品图，得到一批“散图”。
- 商品详情页只采集当前商品左侧主图/轮播图，作为一个商品页分组。
- 用户点击某张图时仍可按原点击采集流程保存。

早期实现如果在保存时再判断图片属于散图还是商品页，会遇到两个问题：

- 用户扫描后可能已经切换页面，保存阶段拿到的“当前页面”不再可靠。
- 前端预览时无法稳定区分“散图平铺”和“商品页文件夹”。

同时，用户需要判断扫描和下载为什么慢、为什么失败，因此需要实时日志。但这类日志主要服务当场排查，不等同于长期审计日志。

## 决策

### 1. 扫描时完成归属标记

每张扫描出的图片在进入图池时就标记：

- `bucket`: `loose` 或 `product`
- `pageKind`: `search`、`detail` 等页面类型
- `groupKey`: 商品页分组键，例如 `temu-g-601101959736135`
- `groupTitle` / `coverUrl`: 前端文件夹标题和封面

保存时只读取这些字段决定目录，不重新推断页面类型。

### 2. 商品详情页只保留主图区域

商品详情页扫描只保留左侧主图/轮播图区域，排除评论、推荐商品、SKU 色块和页面底部图片。

这符合跨境电商采集主图的业务目标，也避免把页面装饰图和推荐商品混进一个商品文件夹。

### 3. 图池 UI 分散图和商品页文件夹

前端图池展示规则：

- 散图：直接平铺显示图片。
- 商品页：显示为文件夹，封面取第一张主图；点进去显示该商品页的全部主图。

这让用户在下载前就能确认图片归属，降低保存后再整理的成本。

### 4. 运行期日志先做内存日志

采集页提供命令行式日志弹窗，复用 `collection:event` / `debug-log` 通道。

日志只保存在当前应用运行期间：

- 最多保留最近 `1000` 条。
- 应用重启后清空。
- 扫描日志显示页面级进度。
- 下载日志显示逐张进度、文件大小、耗时、保存路径或错误原因。

暂不写入 `.workbench/logs/`，避免把高频调试日志和长期审计日志混在一起。

## 影响

### 正面

- 保存目录稳定，不依赖保存时用户停留在哪个页面。
- 前端预览和实际保存目录一致。
- 商品详情页不会保存大量无关小图。
- 下载慢或失败时，用户能从日志看到具体卡在哪一张图。

### 负面

- 如果扫描规则误判，保存阶段不会自动纠正，需要用户清空图池后重新扫描。
- 运行期日志重启后消失，无法用于事后长期追溯。
- 下载仍是串行下载，日志先解决可观察性，不解决速度优化。

## 后续触发条件

如果用户需要事后排查，或需要把日志发给开发者，则新增“导出日志”或写入 `.workbench/logs/collection-{date}.log`。

如果下载速度成为主要瓶颈，则另开 ADR 评估有限并发下载，例如 4-6 张并发，并记录 CDN 限速和失败重试策略。
