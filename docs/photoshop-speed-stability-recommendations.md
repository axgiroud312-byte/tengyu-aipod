## 1. 你们的真实需求（建议必须对齐这个，而不是“通用 PS 脚本”）

腾域 aipod 的 PS 套版不是设计师工具，而是 **跨境 POD 产线中间环节**：

```
印花（02 / 等待套版）
  → PS 套版（智能对象替换 + 导出成品图）
  → 04-上架工作区/{模板批次}/{货号}/01.jpg…
  → 标题 → 店小秘上架
```

因此评价标准不是“功能最全的 mockup 引擎”，而是：

| 目标 | 含义 | 为什么重要 |
|---|---|---|
| **吞吐** | 同样模板 + 同样印花数，单位时间出图更多 | 运营一天要跑大量货号 |
| **可续跑** | 中断后从已完成货号继续，不重做 | 长任务必遇 COM/PS/磁盘抖动 |
| **目录契约** | 输出永远符合上架域结构 | 标题/上架不改代码 |
| **默认够快** | 标准模板无需用户懂 PS 脚本 | 路径 A + 原生切片是主路径 |
| **难模板可兜底** | 300dpi 链接 SO 等少数模板能出正确图 | 路径 B 存在价值，但不应默认 |
| **单机单 PS** | 同一时刻只占一个 Photoshop | 与 invariant「profile/模块互斥」一致；多开 PS 收益低、风险高 |

**不在 v1 需求里的东西（建议明确不做）：**

- 通用可配置编排引擎替换 PS 阶段
- Mac PS 套版（ADR-0007）
- 嵌套/共享 SO 完整递归（ADR-0018 已放到 v1.5）
- 接入 Bulk Mockup 等商业插件作为核心
- 脱离 Photoshop 的自研合成替代复杂 mockup

## 2. 现状判断：你们已经站在正确架构上

### 2.1 当前程序的核心能力（应保留）

| 能力 | 代码/规格落点 | 价值 |
|---|---|---|
| Windows COM + 动态 JSX | `com-adapter` / `com-bridge` / `jsx-generator` | Node 主进程可控，API Key/任务数据不进 PS |
| 路径 A 默认 `placedLayerReplaceContents` | ADR-0018 / spec 05 | 业界公认最快替换路径 |
| 路径 B 可选 `editSmartObject` | ADR-0018 | 兼容 300dpi 链接 SO + fill/fit |
| 模板级 batch：一模板打开一次 | 原生切片 PRD / `generateTemplateBatchJsx` | 去掉“每货号重开大 PSD”的最大浪费 |
| 快路径 history 回退 | JSX 内 `pristineHistoryState` | 同文档连续套版，避免每组 duplicate 整文档 |
| 原生切片优先导出 | PRD + scan `native_slices` | 比 duplicate+crop 更接近插件速度 |
| 跳过已完成（只查文件存在） | 快速路径策略 | 断点续跑低成本 |
| 执行互斥 + 重试 + 取消 | `execution-engine` | 稳定性骨架 |
| 与完整任务/货号契约 | ADR-0015 / CONTEXT | 这是插件和开源脚本没有的护城河 |

### 2.2 相对全网方案的结论

| 外部方案 | 相对你们 | 结论 |
|---|---|---|
| 极简 gist / StackExchange 脚本 | 只有路径 A 循环，无产线 | 内核可参考，不能替代 |
| joonaspaakko Batch Mockup | 嵌套/对齐更强，但偏设计师、偏路径 B | **借鉴算法，不替换引擎** |
| Bulk Mockup 等插件 | 开箱快，但无法嵌入货号/完整任务/工作区 | **不当核心** |
| PS Variables | 对已变换 SO 基本无效 | 不做 |
| UXP 重写 | 投入大，短期不提吞吐 | v1 不做 |

**一句话：继续打磨现有引擎；最大收益在“热路径纪律 + 内存长跑 + 模板规范”，不在换技术栈。**

## 3. 目标热路径（建议定为唯一黄金路径）

```
标准运营模板
  ├─ 印花 SO 可路径 A 替换
  ├─ PSD 内有用户/图层切片（非自动切片）
  ├─ 替换范围优先 topmost / auto
  └─ 印花原样进入 PS（不在套版阶段自动缩放/转码）

执行：
  打开模板 1 次
  → 记录 history 快照
  → 每货号：restore → replaceContents（原图）→ 原生切片导出
  → 每 N 货号：purge histories
  → 关闭模板（不保存）
  → 输出文件已存在则 skip
```

**为什么这是“最好方案”：**

1. **路径 A** 只换内容，保留 mockup 上已做好的透视/变形，单次成本最低。  
2. **同文档循环 + history restore** 避免“每张图复制整份大 PSD”。  
3. **原生切片** 让多视图导出对齐设计师在 PS 里切好的区域，且通常快于 crop 回退。  
4. **skip 文件存在** 让“崩了再跑 / 加跑一批”成本接近零。  
5. **主进程编排** 保证输出目录、完整任务语义、取消与重试，这是产线差异点。  
6. **印花不预处理**：画面可控性优先；尺寸匹配交给模板 SO 与上游导出约定，避免套版隐式改图。

路径 B、无切片 crop、多 SO `top/all` 全部视为 **慢路径/兼容路径**，允许存在，但不应成为默认预期。

## 4. 建议清单（按优先级）

### P0 — 现在就该定规矩（产品 + 运营，代码改动可小）

#### 建议 P0-1：把“模板规范”写成可执行 checklist

**建议内容：**

1. 印花目标层尽量 **嵌入式智能对象**；链接 SO 仅在明确需要时使用。  
2. 默认模板必须能走 **路径 A**；路径 B 仅标注在“特殊模板”上。  
3. 多视图成品必须做 **用户切片或图层切片**；禁止依赖自动切片。  
4. 目标 SO 图层名唯一（推荐 `@印花` 前缀）。  
5. 印花尺寸与 SO 画布的匹配，由**模板制作 + 生图/抠图导出约定**保证，不在套版阶段自动改图。  
6. 去掉无用大图层、隐藏占位层、控制 PSD 体积。  
7. 本机 PS：History States 调低（如 5–20），暂存盘留足空间。

**为什么：**

- 你们已经做了原生切片快路径；**没有切片就会自动回退 crop**，这是“感觉比 WIIMOO 慢”的常见根因，不是引擎写坏了。  
- 路径 B 每张图要进 SO 文档，历史与 IO 都更重；默认开 B 会把黄金路径拖垮。  
- 链接 SO 断链、共享源、嵌套是全网脚本最常见的失败点；你们 v1 本来就不做完整嵌套/共享（ADR-0018）。

#### ~~建议 P0-2：进 PS 前做印花预处理~~（已否决）

> **主理人决策（2026-07-20）：不做。**  
> 理由：套版阶段自动缩放/转码对最终画面可控性差，尺寸策略一偏结果就会偏；印花应原样进 PS，匹配问题交给模板与上游导出约定，而不是套版隐式改图。

曾考虑过的内容（仅作历史记录，**不要实现**）：

- 主进程按 SO bounds 对路径 A 输入图做 cover/fit 缩放、sRGB、格式压缩  
- 预处理产物进 temp，结束后清理  

若未来重新评估，必须先解决：策略可见、可关、可按模板覆盖，且验收矩阵覆盖多种 SO 画布/印花比例——在那之前默认继续 **原样替换**。

#### 建议 P0-2：UI/任务默认强制“快路径画像”

**建议内容：**

| 配置项 | 默认 | 原因 |
|---|---|---|
| 智能对象替换 | `replaceContents`（路径 A） | 最快最稳默认（ADR-0018） |
| 替换范围 | `topmost` 或 `auto` | 完整任务一印花一货号；多 SO 循环更慢 |
| 跳过已完成 | 开 | 续跑刚需 |
| 无原生切片 | 允许跑，但 UI/日志强提示“将明显变慢” | 已有回退；要推动补切片 |
| 路径 B | 仅特殊模板手动开 | 防误用 |

**为什么：**  
产品默认值决定 90% 真实吞吐。功能全不等于默认全开。

### P1 — 下一迭代工程改造（代码应做）

#### 建议 P1-1：模板批处理 JSX 周期性 Purge

**建议内容：**

- 同模板连续处理时，每完成 N 个货号组（建议可配，默认 20–50）执行一次：
  - `app.purge(PurgeTarget.HISTORYCACHES)` 或等价 Action（Histories / All）
- 超大模板或路径 B 任务：N 更小。
- 记录日志：`purge_histories`、`group_index`、可选耗时。

**为什么：**

- Adobe 社区对大批量的一致结论：PS **不会在关文档后把内存还给 OS**，历史与缓存会拖垮长跑。  
- 你们快路径用 history restore 很好，但 **restore ≠ 释放内部缓存**。  
- 比“重写导出”更便宜，是稳定性杠杆。

#### 建议 P1-2：超长任务“软重启 PS + skip 续跑”

**建议内容：**

- 触发条件（任选可配）：运行满 30–60 分钟、完成 K 个模板、连续可恢复 COM 错误达到阈值。  
- 动作：安全结束当前模板 → 释放 COM 桥 → 重启/重连 Photoshop → 继续下一批。  
- 依赖已有“输出文件存在即 skip”，保证不重做出图。

**为什么：**

- 长跑 COM 假死、内存顶满是桌面 PS 自动化常态，不是偶发 bug。  
- 你们已有 skip 与互斥，**重启续跑是架构红利**，应产品化而不是靠人肉重点开始。

#### 建议 P1-3：模板扫描打“快路径资格”标签

**建议内容：** 扫描结果增加只读诊断字段（可先仅日志/UI 提示，不必立刻进 DB 复杂模型）：

| 标签 | 条件 |
|---|---|
| `fast_path_ok` | 有有效原生切片 + 可路径 A + 无未支持嵌套 |
| `slow_export` | 无有效切片 → 将 crop 回退 |
| `needs_path_b` | 链接 SO / 需内部 fill-fit 的模板策略 |
| `unsupported_advanced` | 嵌套/共享需 v1.5 |

任务开始时汇总：本次预计快/慢模板数。

**为什么：**  
把“慢”从事后抱怨变成事前预期，推动模板整改，也方便主理人验收。

#### 建议 P1-4：失败分类与恢复策略表固化

| 错误类 | retryable | 动作 |
|---|---|---|
| COM 忙 / 短暂失败 | 是 | 指数退避（已有） |
| 输出文件缺失 | 是（有限次） | 重跑该组 |
| 找不到 SO 层 | 否 | 标模板坏，不空转重试 |
| 链接源丢失 | 否 | 明确文案：模板链接失效 |
| scratch/磁盘满 | 否 | 停任务，提示清盘 |
| PS 无响应超时 | 特殊 | 杀进程 + 重启 + skip 续跑 |

**为什么：**  
“一律重试 5 次”会把确定性坏模板变成时间黑洞；稳定性 = 该重试才重试。

#### 建议 P1-5：路径 B 护栏

**建议内容：**

- 路径 B 批次默认更小 chunk。  
- 继续保证 `__TENGYU_ARTWORK__` 清理（已有），避免链接源层堆积。  
- 嵌套 SO：失败要明确，禁止半成功导出。  
- 统计 A/B 使用占比（诊断日志即可）。

**为什么：**  
路径 B 是正确性补丁，不是吞吐方案；不加护栏会被当成默认。

### P2 — 有意延后（v1.5 或更后）

| 项 | 为什么延后 |
|---|---|
| 嵌套/共享 SO 完整处理（joonas 级） | 复杂度高；标准 POD 模板应避免 |
| center 与更细对齐 UI | ADR-0018 已界定 |
| UXP 重写引擎 | 不解决当前吞吐主矛盾 |
| 多 Photoshop 实例并行 | 授权/内存/COM 互斥成本高，单机收益差 |
| 无 PS 自研 mockup 渲染 | 复杂置换/混合模式质量不够上架 |

## 5. 明确不建议的做法（以及为什么）

| 做法 | 为什么不要 |
|---|---|
| 默认全部走路径 B | 每张图进 SO，吞吐与稳定性双差 |
| 为追速度去掉 skip/取消/互斥 | 产线会因一次崩溃全盘重做 |
| 引入商业插件当核心 | 无法满足货号/完整任务/工作区 invariant |
| 每货号重新 open 大 PSD | 你们已用模板级 batch，禁止回退 |
| 把自动切片当有效切片 | 碎图/错图；PRD 已否决 |
| 套版阶段自动缩放/转码印花 | 可控性差；尺寸策略隐式，难验收；主理人已否决 |
| 在 02/04 工作区写中间 JSON/脚本 | 违反“业务目录只放业务图片” |
| Mac 并行做 PS | ADR-0007；投入产出比差 |
| 完整任务里为提速破坏“一印花一货号” | ADR-0015 契约，上架/标题会乱 |

## 6. 建议的验收标准（给主理人跑通时用）

用**同一台 Windows 机器、同一组真实模板与印花**对比：

### 6.1 速度基线（建议固定 fixture）

| 场景 | 期望 |
|---|---|
| 路径 A + 有原生切片 + topmost | 主路径；记录“张/分钟” |
| 路径 A + 无切片（crop 回退） | 允许更慢，但日志明确；推动补切片 |
| 路径 B + fill | 正确优先；速度可显著慢于 A |
| 重跑同一任务（skip 开） | 已完成货号接近 0 耗时 |

### 6.2 稳定性基线

| 场景 | 期望 |
|---|---|
| 连续 200+ 货号单模板 | 不无响应；有 purge 时内存曲线更平 |
| 中途取消 | 不写半成品当成功；可再跑续上 |
| 人为制造缺层模板 | 快速失败 + 可读错误，不空转满重试 |
| 完整任务开 PS | 输出目录仍可被标题/上架消费 |

### 6.3 不作为 v1 验收的

- 嵌套三层 SO 全自动  
- 与 Bulk Mockup 逐项功能对标  
- Mac 套版

## 7. 建议落地顺序（实施路线）

```
第 1 步（文档/产品，1 天级）
  └─ 模板 checklist + UI 默认值确认 + 无切片强提示

第 2 步（工程 P1-1 / P1-4，小改动）
  └─ batch JSX 周期性 purge + 失败分类表对齐

第 3 步（工程 P1-2 / P1-3）
  └─ 软重启 PS + 快路径资格标签/汇总

第 4 步（v1.5）
  └─ 嵌套/共享 SO 等能力，按真实模板痛点再开

不做
  └─ 套版阶段印花预处理（自动缩放/转码）— 可控性不足，已否决
```

**为什么这个顺序：**  
先让默认行为与模板规范对齐（零/低代码收益最大）→ 再补长跑 purge（稳定性）→ 再软重启/诊断标签 → 重能力放到 v1.5。**印花原样进 PS，不在套版隐式改图。**

## 8. 和现有文档的关系

| 文档 | 关系 |
|---|---|
| spec/05 | 行为规格仍以 spec 为准；本文只提优化策略 |
| ADR-0007 / 0018 | 不修改；路径 A 默认、路径 B 可选、Windows-only 继续成立 |
| 原生切片 PRD | 方向正确且应坚持；本文补充其“前提条件”（模板切片率、purge）；印花预处理已否决 |
| open-source-references | 外部算法来源；本文明确“借鉴不替换” |
| 若某条 P1 变成强制行为 | 再升格为 ADR 或写回 spec/05 |

## 9. 最终建议（给决策用的三句话）

1. **继续以当前 aipod 引擎为主**，不要换商业插件或重写 UXP；你们已具备产线级优势（目录契约、完整任务、skip、COM、双路径）。  
2. **把“路径 A + 原生切片 + 模板级 batch + skip”定为唯一黄金路径**，路径 B 与 crop 回退只做兼容，并用默认值/提示防止误用。  
3. **下一阶段只打两件事：模板规范、长跑 purge/软重启**（外加默认快路径与诊断标签）；**不做**套版阶段印花预处理——印花原样进 PS，尺寸匹配靠模板与上游约定。

| 指标 | 用途 |
|---|---|
| `ms_open_template` | 模板是否过大 |
| `ms_replace_so` / `replace_mode` | A/B 成本对比 |
| `ms_export` / `export_mode=native|crop` | 切片率是否够 |
| `purge_count` | 长跑是否生效 |

## 附录 B — 运营一页纸（可直接转发）

**想快：**

1. 模板做好用户切片  
2. 用直接替换（路径 A）  
3. 印花 SO 尽量嵌入、图层名唯一  
4. 打开“跳过已完成”  
5. PS 历史记录调低、磁盘留足  

**可以慢但要正确：**

1. 300dpi 链接 SO 再考虑进入内部替换（路径 B）
