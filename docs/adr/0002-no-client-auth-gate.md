# ADR-0002 — v1 不做客户端授权拦截

**状态**：Superseded by [ADR-0011](./0011-customer-login-via-php-auth.md)
**日期**：2026-05-31
**更新**：2026-06-02 需求调整为客户必须登录并获得后台授权后才能使用 Workbench。

---

## 当前结论

本 ADR 保留为历史决策记录，不再作为 v1 实现依据。

新的客户登录与授权门禁以 [ADR-0011](./0011-customer-login-via-php-auth.md) 为准：

- 客户端启动前必须完成客户登录和授权校验。
- 客户登录复用旧 PHP 统一登录体系返回的 `uid + secret`。
- Next 服务端只按 `php_uid` 管理客户授权、到期日和禁用状态，不保存 PHP `secret`。
- 服务端仍不接触用户图片、API Key、任务数据、生图任务。

## 历史背景

当前产品的核心价值在本地桌面工作台：采集、生图、检测、PS 套版、标题和上架。用户图片、API Key、任务记录和工作流都在客户端本机。

旧方案把客户端可用性和授权状态绑定，会增加启动失败、售后排障和本地状态不一致问题，也会干扰主理人当前优先级：先把完整业务链路跑顺。

## 历史决策（已废弃）

v1 不实现客户端授权门禁：

- 客户端启动、选择工作区、保存 API Key、拉取 Skill、运行业务模块都不需要客户端授权 token。
- 服务端 `/api/skills` 公开给客户端读取启用的 Skill。
- Admin 后台保留客户记录、Skill、公告和版本；客户封号只作为后台记录状态。
- 不采集硬件信息，不绑定本机，不做设备席位。
- 后续如果要重新引入登录、订阅、席位或授权，需要新增 ADR 和独立实现，不复用旧链路。

## 历史影响

正向影响：

- 首次启动更短，用户只需要选择工作区和填写本地 API Key。
- 离线或服务端不可用时，客户端仍能使用本地 Skill 缓存和本地配置。
- 服务端边界更清楚：不接触图片、API Key、任务数据和客户端业务状态。
- 删除旧授权相关表、IPC、前端状态、轮询器和 API，减少故障点。

代价：

- v1 不提供服务端强管控的客户准入。
- 客户管理后台只做记录，不控制客户端运行。
- 商业化授权需要后续重新设计。

## 历史实现

- 客户端：`packages/client/src/main/onboarding.ts` 只处理首次设置和 API Key 保存。
- 本地状态：`setup-state.json` 记录首次设置完成时间。
- 服务端：`packages/server/src/app/api/skills/` 不要求客户端 token。
- 数据库：Prisma schema 不保留旧授权表。

## 历史不做项

- 不做硬件绑定。
- 不做客户端 token 轮询。
- 不做右上角授权状态徽章。
- 不做服务端代理生图、LLM 或用户 API Key。
