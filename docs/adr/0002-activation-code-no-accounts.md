# ADR-0002 — 用激活码模式，不引入用户账号系统

**状态**：已采纳
**日期**：2026-05-23

## 背景

腾域 aipod 需要一种"付费 → 软件可用"的授权机制。常见做法：

1. **用户账号 + 订阅**：用户注册邮箱 + 密码，付费后开通订阅
2. **激活码**：管理员后台生成激活码，用户付款后填码即可
3. **硬件加密狗**：物理硬件绑定

支付能力 v1 完全不做（线下转账），是后台手动开通模式。

## 候选方案与决策

### 选项 A：账号 + 订阅（最常见）

需要：
- User 表 + 邮箱验证流 + 密码加密
- Subscription 表 + 续费机制
- Auth.js 或自建鉴权
- 用户找回密码、修改邮箱、设备管理 UI

工作量大，且对 v1 的小用户群（50-100 人）来说**用户体验差**（要注册、记密码）。

### 选项 B：激活码（**采纳**）

需要：
- ActivationCode 表（含天数、设备数、归属客户）
- DeviceActivation 表（指纹绑定）
- 单一 `/api/activate` 端点
- 不需要邮箱验证、密码加密、找回流程

### 选项 C：硬件加密狗

完全不考虑（成本高、用户接受度差）。

## 决策

**采用激活码模式（选项 B）。**

理由：

1. **用户体验最简**：用户付款后拿到一串码，粘贴即用，不需要注册任何账号、记任何密码。

2. **运维最简**：管理员后台批量生成激活码 + 发码即可，不需要管理用户账号。

3. **支付灵活**：v1 完全线下支付（微信/支付宝转账给你），未来要接 Stripe/Lemon Squeezy 也只是把"线下转账"换成"自助生成激活码"。

4. **数据库轻量**：3 张核心表（Customer / ActivationCode / DeviceActivation）vs 账号模式的 5+ 张表。

5. **隐私友好**：不强制收集用户邮箱手机号。可作为卖点。

6. **数据迁移容易**：未来如果改用账号模式，激活码可以平滑升级（一个激活码 = 一个新账号）。

## 影响

### 正面

- 用户首次启动只需输入一串码
- 你后台的工作量极小（创建码 / 改天数 / 解绑设备 / 封号）
- 数据隐私好（甚至匿名码完全不收集用户信息）

### 负面

- 用户丢码需要找你重发（找客户唯一靠的是手机号 / 微信）
- 不能搞"邮件 promotion"（没有用户邮箱）
- 不能搞"自助续费"（用户必须找你）

### 缓解措施

- 客户端 [设置 → 激活] 页能看到完整码（用户能自己复制保存）
- 客户购买时强制填手机号 + 备注（admin 后台能查回）
- 匿名码也有 batch_id，找你时按"我是从哪个批次拿的码"也能定位

## 子账号问题

用户购买"主账号 + N 个子账号"实际等于**一个激活码 + 多设备槽位**。不引入"子账号独立登录"概念（保持简单）。

如果未来出现"团队多人协作各自数据"的需求，再升级到账号系统。

## 替代决策的触发条件

考虑升级到账号系统的信号：
- 用户量 > 1000，码管理成本变高
- 用户强烈要求"邮件接收激活码"、"自助续费"
- 想做"用户社区"、"分享 Skill"等需要登录的功能
