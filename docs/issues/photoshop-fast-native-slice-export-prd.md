# PRD: PS 套版快速原生切片导出

## Problem Statement

当前 PS 套版比现有 WIIMOO 插件慢很多。用户的实际需求很窄：批量替换 PSD 模板里的智能对象，然后把已经在 PSD 中做好的 Photoshop 原生切片导出为成品图。当前实现每个套版组会复制一整份 PSD 文档，再裁切、保存、关闭；同时还会做 hash、数据库记录、细粒度日志和输出校验。这些工程化动作提高了可追踪性，但对“只替换智能对象并导出切片”的日常 POD 套版来说成本过高。

用户希望默认套版路径更接近 Photoshop 插件式批处理：每个模板打开一次，在同一文档里连续替换智能对象，优先用 Photoshop 原生切片导出，最终产物仍保持当前 `04-上架工作区` 目录结构，后续标题和上架模块不需要改。

## Solution

PS 套版默认改为快速原生切片导出路径：

1. 每个 PSD 模板只打开一次。
2. 在同一个 PSD 文档中按套版组连续执行智能对象替换。
3. 智能对象替换范围沿用现有 `topmost / auto / top / all` 逻辑。
4. 预检 PSD 是否存在有效 PS 原生切片；只认用户切片和图层切片，不把自动切片当作有效切片。
5. 有有效 PS 原生切片时，使用 Photoshop Web 导出切片到临时目录。
6. 导出的切片按顺序重命名/移动到当前输出结构。
7. 无有效 PS 原生切片时，不阻塞用户，在任务日志提示“将使用旧模式裁切导出，速度会慢一些”，并自动回退旧裁切导出。
8. “跳过已完成”改为轻量判断：只检查目标输出文件是否都存在，不查数据库、不算 hash。
9. 独立 PS 套版和完整任务 PS 套版阶段全部默认使用这一路径。

## User Stories

1. As a 跨境电商运营, I want PS 套版默认更快, so that 批量处理 PSD 模板时不用等待当前慢路径。
2. As a 跨境电商运营, I want 一个 PSD 模板只打开一次, so that Photoshop 不会为每张印花重复加载大模板。
3. As a 跨境电商运营, I want 每张印花在同一个模板文档里连续替换智能对象, so that 套版速度接近常见 PS 插件。
4. As a 跨境电商运营, I want 程序优先使用我在 PSD 中做好的原生切片, so that 导出区域和我在 Photoshop 里设计的一致。
5. As a 跨境电商运营, I want 自动切片不被误当成有效切片, so that 不会导出一堆 Photoshop 自动填充的无效碎图。
6. As a 跨境电商运营, I want 切片导出后仍按当前货号文件夹和文件名保存, so that 标题生成和上架模块可以继续读取。
7. As a 跨境电商运营, I want 独立 PS 套版页默认走快速路径, so that 手动套版时直接得到速度提升。
8. As a 跨境电商运营, I want 完整任务里的 PS 套版阶段也默认走快速路径, so that 端到端流程不会在套版阶段重新变慢。
9. As a 跨境电商运营, I want 没有原生切片的模板仍能跑完, so that 旧模板不会因为新快速路径而失效。
10. As a 跨境电商运营, I want 没有原生切片时看到日志提示, so that 我知道这次为什么会慢。
11. As a 跨境电商运营, I want 已完成目标文件存在时直接跳过, so that 重跑任务时不用重复导出已经存在的货号。
12. As a 跨境电商运营, I want 跳过判断不要重新 hash 大图, so that 重跑预检也足够快。
13. As a 跨境电商运营, I want 多智能对象模板仍沿用原来的替换范围选项, so that 现有 `topmost / top / all` 工作方式不变。
14. As a 跨境电商运营, I want PSD 扫描结果继续缓存, so that 同一个模板后续不需要反复完整扫描。
15. As a 开发者, I want PSD 扫描能区分有效原生切片和自动切片, so that runner 能稳定选择快速或回退路径。
16. As a 开发者, I want 快速 JSX 生成逻辑可以单测验证, so that 不会意外恢复每张印花复制模板的慢路径。
17. As a 开发者, I want 批处理 runner 在同一个高层接缝验证快速路径和回退路径, so that 行为测试贴近真实套版入口。
18. As a 开发者, I want 输出目录结构保持不变, so that 标题、上架和结果预览不需要级联重构。
19. As a 开发者, I want 旧裁切导出只作为自动回退, so that 用户不用理解两套模式开关。
20. As a 开发者, I want 真实 PSD 测速成为验收项, so that 这次改造真的解决慢的问题。

## Implementation Decisions

- 默认执行路径改为快速原生切片导出，覆盖独立 PS 套版和完整任务 PS 套版阶段。
- 不新增用户可见的“快速/兼容”模式开关；旧裁切导出只作为无有效原生切片时的自动回退。
- PSD 扫描继续保留并缓存；扫描结果需要包含有效 PS 原生切片信息，且能排除自动切片。
- 有有效 PS 原生切片时，批处理 JSX 打开模板一次，循环处理套版组，在同一个文档里替换智能对象，并通过 Photoshop Web 导出切片到临时目录。
- Web 导出的切片不直接使用 Photoshop 生成文件名；主进程或 JSX 后处理必须按切片顺序整理为现有输出文件名。
- 输出布局保持现状：独立 PS 套版保留单次套版批次下的货号文件夹，完整任务和模板优先布局保留模板批次下的货号文件夹。
- 智能对象选择范围不变，继续沿用 `topmost / auto / top / all`。
- 智能对象替换方式继续沿用 `replaceContents / editSmartObject`，快速路径主要优化文档生命周期和切片导出方式。
- “跳过已完成”在快速路径中只检查目标输出文件存在，不再依赖 workflow step、artifact hash 或数据库记录。
- 运行日志保留任务级关键事件：模板开始、检测到原生切片、无切片回退、组完成、任务完成、失败。细粒度逐文件 hash 和数据库 artifact 记录不作为快速路径要求。
- 无有效原生切片时不阻塞用户，记录一次提示并自动走旧裁切导出。
- 原生切片导出失败时应暴露明确错误；是否回退旧裁切导出由实现阶段根据可恢复性分类，但不得静默吞掉真实失败原因。

## Testing Decisions

- 最高层测试接缝是 PS 批处理 runner 的外部行为：给定印花、模板扫描结果和输出配置，验证快速路径或回退路径的选择、输出结构和结果统计。
- PSD 扫描测试只验证用户切片/图层切片可识别、自动切片被排除、扫描结果可缓存。
- JSX 生成测试只锁定关键行为：模板打开一次、循环替换、使用切片/Web 导出、没有每张印花复制模板的 `baseDocument.duplicate()` 慢路径。
- 跳过已完成测试验证只检查目标输出文件存在，不查数据库、不计算 hash。
- 完整任务测试验证 PS 阶段传入配置后默认走同一快速套版 runner，并保持等待套版印花名到货号文件夹名的映射。
- 真实验收必须用至少一个带用户切片的真实 PSD、同一批印花，对比改造前后的耗时、输出张数、文件命名和上架读取。
- 无切片真实或 fixture 模板必须验证日志提示和旧裁切回退。

## Out of Scope

- 不做 Mac PS 套版支持。
- 不改变 `04-上架工作区` 输出结构。
- 不改标题生成和上架模块读取规则。
- 不新增通用可配置编排引擎。
- 不实现手动切片编辑 UI。
- 不把自动切片作为有效导出区域。
- 不要求所有旧模板都具备原生切片；无切片模板走旧裁切回退。
- 不追求本次彻底解决所有嵌套智能对象、共享智能对象和特殊 PSD 结构问题。

## Further Notes

- 新术语“PS 原生切片”已区别于现有“裁切区域”。裁切区域是腾域从参考线或图层 bounds 推导出的导出区域；PS 原生切片是 PSD 内由用户或图层创建、可由 Photoshop Web 导出流程识别的切片。
- 当前慢点主要来自每个套版组复制整份 PSD 文档，以及完成后 hash/数据库记录等工程化成本。快速路径的核心是减少 Photoshop 文档复制和主进程后处理。
- 这份 PRD 标签应为 `ready-for-agent`。
