# ADR-0006 — 统一生图模块按"业务能力"分 4 Tab，不按 provider 分

**状态**：已采纳
**日期**：2026-05-23

## 背景

腾域有两个生图 provider：
- `comfyui-chenyu`（晨羽智云上的 ComfyUI）
- `grsai`（付费模型 API）

且有 4 个生图能力：
- 文生图
- 图生图
- 提取
- 抠图

且每个能力可能由不同 provider 实现，且实现方式（直接喂图 vs 喂 LLM 看图）也不同。

如何组织 UI 和代码？

## 候选方案

### 方案 X：按 provider 分 2 个独立模块（最初想法）

```
生图-comfyui 模块（4 个子能力 Tab）
生图-中转站 模块（4 个子能力 Tab）
```

- ✅ 清晰区分 provider
- ❌ 用户视角错乱："我要图生图，要进哪个模块？"
- ❌ 重复实现（4 个 Tab 实现两遍）
- ❌ 模块边界混乱（中转站的"提取"实际是图生图特化）

### 方案 Y（采纳）：按能力分 4 Tab，每 Tab 内选 provider

```
统一生图模块
├─ 文生图 Tab
│   实现方式：● Grsai 付费  ○ ComfyUI 免费（不可用）
│
├─ 图生图 Tab
│   实现方式：● Grsai 付费  ○ ComfyUI 免费
│
├─ 提取 Tab
│   实现方式：● Grsai 付费  ○ ComfyUI 免费
│
└─ 抠图 Tab
    实现方式：● ComfyUI 直接抠图  ○ 付费生黑白图 + ComfyUI 混合
```

- ✅ 用户视角直接（"我要做什么" → "用什么实现"）
- ✅ UI 一次实现，多 provider 复用
- ✅ 模块边界清晰

## 决策

**采用方案 Y**。

### 实现矩阵

| 能力 | comfyui-chenyu | grsai |
|---|---|---|
| 文生图 | ❌（ComfyUI 不做文生图）| ✅ |
| 图生图 | ✅（多个工作流）| ✅（5 种生成方式）|
| 提取 | ✅（提取工作流）| ✅（图生图 + 提取 skill）|
| 抠图 | ✅（抠图工作流 / 混合路径）| ❌（不内置）|

### 输出域统一

```
02-生图/
├─ 01-文生图/{印花ID}.png       ← provider 信息在数据库，不分目录
├─ 02-图生图/{印花ID}_v1.png
├─ 03-提取/{印花ID}.png
└─ 04-抠图/{印花ID}.png         ← 最终透明底
```

**目录不再按 provider 分**。一个印花的 provider 来源在数据库 `artifacts.provider` 字段。

### 各能力的 provider 实现细节

| 能力 | comfyui 实现 | grsai 实现 |
|---|---|---|
| 文生图 | - | 直接 `POST /v1/api/generate` |
| 图生图 | 工作流接受参考图 | 视觉 LLM 看参考图 → 写提示词 → 文生图（5 种模式：纯文字/参考构图/参考风格/构图+风格/自己写） |
| 提取 | 专门的提取工作流 | 图生图 + 云端"提取 skill" prompt |
| 抠图 | 抠图工作流 OR "付费生黑白图 + ComfyUI 转遮罩 + 图像混合" 混合路径 | - |

## 抠图特殊处理

**付费模型抠图效果差**（实际验证）→ 没有"纯付费抠图"路径，要不走 comfyui 直接抠图工作流，要不走混合路径（付费生黑白图，再让 ComfyUI 做遮罩混合）。

## 影响

### 正面

- UI 跟用户心智模型对齐（按"做什么"分）
- 代码层 `provider abstraction` 可以扩展（未来加 Replicate / 自部署 ComfyUI 等）
- 数据库 artifact 表的 step 字段对应能力（不是 provider），统计和查询友好

### 负面

- 同一能力的两种 provider 实现可能产出风格不同，用户要理解差异
- "抠图无纯付费"对部分用户不直观（需要 UI 上明确说明）

## 与方案 X 的兼容性

如果未来用户量大且**确实希望按 provider 切换全模块**，可以在 UI 层额外加一个"provider 模式切换"（全局切换默认 provider），但**业务模型不变**——仍是 4 能力，只是默认 provider 切换。

## 替代决策的触发条件

- 出现第三个生图 provider 且与 grsai / comfyui 行为差异极大（v1.5）
- 用户调研发现 80%+ 用户只用一个 provider，"切换"反而成本

那时可以考虑"全局 provider 模式" + "高级用户每 Tab 切换"。
