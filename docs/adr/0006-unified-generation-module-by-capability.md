# ADR-0006 — 统一生图模块按"业务入口"分 Tab，不按 provider 分

**状态**：已采纳
**日期**：2026-05-23

## 背景

腾域有两个生图 provider：
- `comfyui-chenyu`（晨羽智云上的 ComfyUI）
- `grsai`（付费模型 API）

基础生图能力包括：
- 文生图
- 图生图
- 提取
- 抠图

当前 UI 另外提供一个 ComfyUI-only 的组合入口：提取后抠图（提取工作流 → 抠图工作流）。

且每个能力可能由不同 provider 实现，且实现方式（直接喂图 vs 喂 LLM 看图）也不同。

如何组织 UI 和代码？

## 候选方案

### 方案 X：按 provider 分 2 个独立模块（最初想法）

```
生图-comfyui 模块（4 个子能力 Tab）
生图-中转站 模块（4 个子能力 Tab）
```

- ✅ 清晰区分 provider
- ❌ 用户视角错乱："我要图生图，要进哪个模块？"
- ❌ 重复实现（多个入口实现两遍）
- ❌ 模块边界混乱（中转站的"提取"实际是图生图特化）

### 方案 Y（采纳）：按业务入口分 Tab，在入口内组织运行路径

```
统一生图模块
├─ 文生图 Tab
│   统一页：提示词生成/审稿共用；右侧生图路径选择 Grsai 或 ComfyUI 工作流
│
├─ 图生图 Tab
│   保持 Grsai / ComfyUI 原入口，不合并
│
├─ 提取 Tab
│   实现方式：● Grsai 付费  ○ ComfyUI 免费
│
├─ 抠图 Tab
│   实现方式：● ComfyUI 直接抠图  ○ 付费生黑白图 + ComfyUI 混合
│
└─ 提取后抠图 Tab
    实现方式：● ComfyUI 提取工作流 → ComfyUI 抠图工作流
```

- ✅ 用户视角直接（"我要做什么" → "用什么实现"）
- ✅ UI 一次实现，多 provider 复用
- ✅ 模块边界清晰

## 决策

**采用方案 Y**。

当前 UI 收敛形态：只合并文生图。图生图保持原来的 Grsai / ComfyUI 分流，因为两者输入心智不同：Grsai 图生图围绕“参考图 + 提示词审稿 + 是否把参考图发给生图模型”，ComfyUI 图生图围绕“已登记印花 artifact + 本地工作流”。提取后抠图作为组合入口单独展示，方便用户从源图直接得到透明底结果。

生图页提供统一的运行期日志入口，不按 provider 拆分。日志按钮位于生图模块顶部，弹窗以命令行形式显示提示词生成、任务提交、模型调用进度、完成/失败和保存路径。该运行期日志只保存在前端内存中，最多最近 `1000` 条。另有统一诊断日志写入 `.workbench/logs/diagnostics/generation/{taskIdOrRunId}.jsonl`，用于排查 LLM / provider 的完整请求参数、原始返回、轮询/重试次数和解析失败；不记录 API Key 或图片 base64 原文。

提取能力共用同一套提取 Skill。用户在云端可配置多条提取提示词，客户端在 Grsai 提取和 ComfyUI 提取里都从这组 Skill 中选择；一张源图对应一次运行，provider 差异只体现在 Grsai 图生图接口或 ComfyUI 提取工作流。

### 实现矩阵

| 能力 | comfyui-chenyu | grsai |
|---|---|---|
| 文生图 | ✅（文生图工作流）| ✅ |
| 图生图 | ✅（多个工作流）| ✅（4 种模式；“生图时带参考图”默认关闭）|
| 提取 | ✅（提取工作流 + 提取 Skill）| ✅（图生图 + 同一套提取 Skill）|
| 抠图 | ✅（抠图工作流 / 混合路径）| ❌（不内置）|
| 提取后抠图 | ✅（提取工作流 → 抠图工作流）| ❌ |

### 输出域统一

```
02-印花工作区/
├─ 文生图/{taskId}/{印花ID}.png       ← provider 信息在数据库，不分目录
├─ 图生图/{taskId}/{印花ID}_v1.png
├─ 提取/{taskId}/{印花ID}.png
└─ 抠图/{taskId}/{印花ID}.png         ← 抠图 / 提取后抠图的最终透明底
```

**目录不再按 provider 分**。能力目录下按任务建子文件夹，provider 来源在数据库 `artifacts.provider` 字段。

### 各能力的 provider 实现细节

| 能力 | comfyui 实现 | grsai 实现 |
|---|---|---|
| 文生图 | 文生图工作流 | 直接 `POST /v1/api/generate` |
| 图生图 | 工作流接受参考图 | 视觉 LLM 看参考图 → 写提示词 → 生图；“生图时带参考图”默认关闭，可由用户开启（4 种模式：参考构图/参考风格/构图+风格/自己写） |
| 提取 | 专门的提取工作流 + 云端"提取 Skill" prompt | 图生图 + 同一套云端"提取 Skill" prompt |
| 抠图 | 抠图工作流 OR "付费生黑白图 + ComfyUI 转遮罩 + 图像混合" 混合路径 | - |
| 提取后抠图 | 提取工作流输出到临时目录，再交给抠图工作流；只保留最终透明底 | - |

## 抠图特殊处理

**付费模型抠图效果差**（实际验证）→ 没有"纯付费抠图"路径，要不走 comfyui 直接抠图工作流，要不走混合路径（付费生黑白图，再让 ComfyUI 做遮罩混合）。

## 影响

### 正面

- UI 跟用户心智模型对齐（按"做什么"分）
- 代码层 `provider abstraction` 可以扩展（未来加 Replicate / 自部署 ComfyUI 等）
- 数据库 artifact 表的 step 字段对应能力（不是 provider），统计和查询友好

### 负面

- 同一能力的两种 provider 实现可能产出风格不同，用户要理解差异
- "抠图无纯付费"对部分用户不直观（需要 UI 上明确说明）

## 与方案 X 的兼容性

如果未来用户量大且**确实希望按 provider 切换全模块**，可以在 UI 层额外加一个"provider 模式切换"（全局切换默认 provider），但**业务模型不变**——仍是 4 能力，只是默认 provider 切换。

## 替代决策的触发条件

- 出现第三个生图 provider 且与 grsai / comfyui 行为差异极大（v1.5）
- 用户调研发现 80%+ 用户只用一个 provider，"切换"反而成本

那时可以考虑"全局 provider 模式" + "高级用户每 Tab 切换"。
